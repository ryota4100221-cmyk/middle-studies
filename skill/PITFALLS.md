## 7. `transform_apply(scale=True)` が location を 0 に落とす（003で半日溶かした罠）
```python
bpy.ops.mesh.primitive_cube_add(size=1, location=(0,0,z))
o = bpy.context.active_object
o.scale = (sx, sy, sz)
bpy.ops.object.transform_apply(scale=True)   # ← ここで o.location.z が 0 に化ける
```
**症状**: 生成物が全部 z=0（床）に落ちる。ただし直後に `keyframe_insert(location)` で毎フレーム座標を書き込むオブジェクトは、キーが上書きするので**気づかない**。アニメを打たない固定オブジェクト（003では断層のライム層）だけが床に取り残される。
**回避**: `transform_apply` の直後に `o.location = (0.0, 0.0, z)` を明示再設定する。あるいは scale を transform_apply せず `o.dimensions` / 親スケールで持つ。
**教訓**: 「固定オブジェクトだけ位置がおかしい」時はまず transform_apply を疑う。デバッグは `o.matrix_world.translation` を print して数値で当たりを取ると速い（テストレンダーを何周も回すより桁違いに速い）。

### 7-b. 逆の罠：`--factory-startup` では location=loc 再設定が二重オフセットになる（004で確認）
2026-07-11の004（ANDON）で判明。**`--factory-startup` 付き**で `transform_apply(scale=True)` を呼ぶと、
Blenderは**メッシュ頂点をワールド位置に焼き込み、`location` を 0 に落とす**（=見た目の位置は変わらない）。
この状態は**すでに正しい位置**なので、直後に `o.location = loc` を足すと **loc ぶん二重に上がる**（004では籠が2倍の高さに浮いた）。

```python
bpy.ops.mesh.primitive_cube_add(size=1, location=(x,y,z))
o.scale = (sx, sy, sz)
bpy.ops.object.transform_apply(scale=True)   # ← ここでメッシュがworld焼き込み・location→0（位置は既に正しい）
# o.location = (x,y,z)   # ✗ --factory-startup下ではこれで z が 2z になる
```

**#7と#7-bの使い分け**:
- 固定オブジェクトが**床に落ちた**（z→0）→ #7。`o.location=loc` を足して復旧。
- 固定オブジェクトが**倍の高さに浮いた**（z→2z）→ #7-b。`o.location=loc` を**消す**。
- **確実な見分け方**: 造形直後に `probe` スクリプトで `o.matrix_world.translation.z` と `bound_box` のworld z範囲を print し、`location=loc` の有無で挙動を2秒で確定させてから本レンダーに進む（004ではこれで即断できた）。挙動は起動オプション（`--factory-startup`）やBlenderバージョンで割れるので、**毎回probeで確認**するのが最速かつ確実。

## 9. 回転リグは「原点」に置く。CENTER_Zに置いて matrix_parent_inverse で相殺しない（005で確認）
005（KAWA）で、リグを `empty_add(location=(0,0,CENTER_Z))` に置き、子の `matrix_parent_inverse = rig.matrix_world.inverted()` で相殺しようとしたら、
**メッシュをz≈0で生成したパーツ（皮）と、primitiveのlocationでCENTER_Zに生成したパーツ（餡）が別々の高さに割れた**（皮が床、餡が中空）。
matrix_parent_inverse はリグのオフセットを打ち消すので、生成時の絶対zがそのまま残り、パーツごとに基準がズレる。

**正解（004/005で確立した方式）**:
- リグ（回転用Empty）は必ず**原点 `(0,0,0)`** に置く。
- 全パーツを**ワールドの CENTER_Z に配置**する（メッシュをz≈0で作ったら `o.location=(0,0,CENTER_Z)`、primitiveなら `location=(0,0,CENTER_Z)` で作る）。
- 親子付けは `o.parent = rig` だけ（matrix_parent_inverse は触らない＝既定のidentity）。
- x=y=0 の中心軸上にあるので、リグを世界Z軸まわりに回せばオブジェクトは自分の垂直軸で回転する。

## 10. 環状/放射メッシュ＋Bevelは「扇状ストリーク」を生む。angle_limitで鋭角のみ面取り（013で確認）
013（HAGURUMA）で、歯車の環状面を内周→外周の1本の長いクアッドで張り、Bevelモジュール（既定=全エッジ）を掛けたら、
**平面上の放射エッジ全部が面取りされ、高解像度レンダーで中心から広がる「扇状ストリーク」**が出た（低解像度テストでは見えず、heroサイズで初めて発覚）。

**回避**:
- Bevelは `bev.limit_method='ANGLE'` ＋ `bev.angle_limit=radians(30)` にして、**鋭角の辺（歯先・縁・穴）だけ**面取り。平面上の放射エッジ（二面角ほぼ0）は除外される。
- 環状面は**半径方向を複数リングに分割**して、内→外の長い1本クアッドを避ける（面が素直になり陰影も安定）。
- bmeshで大量頂点を作った後は `bmesh.ops.remove_doubles` ＋ `recalc_face_normals` で法線の乱れ（これもストリークの原因）を潰す。
- **教訓**: 手続き造形は必ず一度**heroサイズ（1600×2000相当）でテスト**する。480pxテストでは面のアーティファクトが潰れて見えない。

## 11. 均一な暗い面の上の「集中した発光点」はGlare(BLOOM)が箱状＋十字アーティファクトを出す（011で確認）
011（NAMI）で、黒い水面の中心に小さな発光球＋点光源を置いたら、中心に**淡い正方形の箱＋十字（＋）**が出た。
- **消えない対処**（全部ハズレ）：マテリアルをマット化（反射除去）／発光体の`visible_shadow=False`／fillライトの`use_shadow=False`／発光輝度を下げる——どれも効かなかった。
- **正体**：シリーズ不変のコンポジタGlare（Type=BLOOM）はFFT畳み込みなので、**強く集中した輝点の周りに正方形の畳み込みカーネル跡＋streakの十字**を残す。004/007等では核が大きい／背景が忙しく目立たないが、**均一な暗い面（水面・大きな黒盤）の上だと露出する**。
- **効いた対処**：中心の**集中発光オブジェクトを廃止**する。輝点を作らなければFFT箱も十字も出ない。011では「中心の芯」をやめ、最内の生まれたてリング＋`shadow_soft_size`大の柔らかい点光源のみにして解消。
- **教訓**：大きな均一面（水面/黒盤/床）の上に強い輝点を置く構図は避ける。中心の光は「面で」表現するか、リング/エッジなど非集中の形にする。Glare設定は不変なので**造形側で輝点を作らない**のが正解。

## 12. 「伸びて千切れる」液滴はメタボール2球ではなく回転体＋シェイプキーで作る（018で確認）
018（SHIZUKU 雫）で、黒い液滴の伸展→ネック→千切れを**メタボール2球**（上下2オブジェクトを同名グループで融合）で作ろうとして破綻した。
- **症状**: 2球はごく近距離でしか融合せず、少し離すと即座に**独立した2つの球**にスナップ分離する（ネック結合レンジが短い）。しかも間に置いた発光ライム球が**裸の緑玉**として宙に浮き、「等サイズの黒2球＋緑の玉」という完全にオフブランドな絵になった。低threshold/高stiffness/travel縮小でも、ドラマチックな長い細ネックは出せない。
- **回避（正解）**: 液滴を**surface of revolution（回転体）としてラーゼ生成**し、ネックを痩せさせる**シェイプキー "stretch"** を1枚足して伸展量svでアニメする。黒メッシュは**繋がったまま**細い糸になるので、ネックに置いた発光ライム球が「痩せた黒糸の断面＝芯」として読める（裸の緑玉にならない）。sv=0.5(1-cos2πt)で完全ループ。ライム球はsvに比例して露出（融合時sv=0で半径0＝球に戻ると芯が消える）。
- **造形メモ**: 回転体プロファイルを対称レモン（r=√sin(πu)）にするとネックの上側が**傘/きのこ状に膨らむ**。頂点(u=0)細・底(u=1)太の**ティアドロップにskew**（例 env=√sin(πu)・skew=0.58+0.62u）し、ネック位置UWを付け根寄り(0.19)に上げると小さなnub＋細ネック＝ちゃんと"雫の垂れ"に読める。頂点/底のポール中心をシェイプ間で頂点数一致させる（両シェイプ同数リング＋2ポール）。
- **教訓**: メタボールは「融合したまま呼吸/回転」（004行灯の核など）には向くが、「伸びて千切れる」トポロジー変化の演出には結合レンジが足りない。連続した細ネックの造形は回転体＋シェイプキーが確実。

## 13. 露出した発光体は Base Color=LIME にすると白飛びする。純発光体として組む（020で確認）
020（TSUGITE）で、黒柱から露出する発光ライムのほぞを `Base Color=LIME + Emission=LIME/2.8` で作ったら**中心が白緑に飛んだ**。
- **正体**: ほぞは柱の外に露出していて **key(1400W) を遮る物が無い**。LIMEの拡散面がキーの白色光をまともに反射し、その白が発光に上乗せされる。001の餡や019の裏面パネルが飛ばないのは、**他のパーツに遮られて直接キーが当たらない**から。
- **回避**: 露出する発光体は**純発光体として組む**。`Base Color` を暗いライム（例 `(0.015,0.030,0.005)`）に落として反射成分を消し、色は emission だけで決める。`Specular IOR Level` も 0.10 程度に。
- **教訓**: 「発光体が周りに遮られているか、裸で露出しているか」でマテリアルの作り方を変える。裸なら純発光体。

## 14. 発光の色は**「中間調」をピクセル値で合わせ、勾配（ホットコア＋ハロー）を必ず残す**（020で確立・2026-07-21に全面改訂）
> ⚠️ **旧版の「ライム平均を#A5E02Eに一致・白飛び0%」は誤りだった。** その基準で作った016/020〜026は、指標上は毎作合格しながらライムが**縁のくっきりしたベタ塗り（ペンキ・シール）**に退化した（#24参照）。光が光に見えるのは「白く飛んだ芯 → #A5E02E → 暗部」という**勾配**があるからで、「平均=目標・白飛び0%」の最適化は数学的必然としてベタ塗りに収束する。001 THE FILLING（旧基準で「飛び気味」と欠点扱い）こそシリーズ最良の光だった。

- **正しい手順**: ライム画素を明るさで並べ、**中間調（20〜80パーセンタイル帯の平均）を `#A5E02E` に寄せる**。上位5〜10%は白に飛んでよい——**むしろ飛ぶべき**（それがホットコア）。Bloom閾値1.2を超える画素が無いとハローは物理的に出ない。
- **合格ラインの実測値**（heroで測る）:
  - ライム画素の**輝度std ≥ 35**（40〜66が旧作の健全域。11〜16はペンキ）
  - **ハロー画素**（淡い明るめの緑 `150<R<230, G>200, 90<B<190, G>B+30`）**が1,000以上**（旧作は数千〜数万。0はブルーム死）
  - 中間調が `#B0E850`〜`#D5FFA8` 域（**2026-07-21に手本4作の実測で較正**：001 `#D1FD98` / 002 `#CFF39F` / 013 `#B4EE66` / 007 `#D1FDAB`。発光の中間調はブランド色 `#A5E02E` より**明るい側**に来るのが健全。`#A5E02E` ぴったりに合わせるとペンキになる——それが旧版の間違いだった）
```python
from PIL import Image
px = list(Image.open('hero.png').convert('RGB').getdata())
lime = [p for p in px if p[1] > 90 and p[1] > p[2] + 45 and p[1] >= p[0] + 20]
lum = sorted((0.2126*p[0]+0.7152*p[1]+0.0722*p[2], p) for p in lime)
mid = [p for _, p in lum[len(lum)//5 : len(lum)*4//5]]          # 20〜80%帯
avg = tuple(round(sum(p[i] for p in mid)/len(mid)) for i in range(3))
m = sum(l for l,_ in lum)/len(lum)
std = (sum((l-m)**2 for l,_ in lum)/len(lum))**0.5
halo = sum(1 for p in px if 150<p[0]<230 and p[1]>200 and 90<p[2]<190 and p[1]>p[2]+30)
print("mid %s  std %.1f  halo %d" % ("#%02X%02X%02X"%avg, std, halo))
```
- **シリーズの手本**: 002 OBI（std 45・halo 2万）/ 001 THE FILLING（std 42・halo 6万）/ 013 HAGURUMA（std 55）。**迷ったら002と並べる。**
- **例外（平面の純発光体）**: 正対する平面・均一な発光面（020のほぞ・025の眼など）は面内が均一なので **std が低く出る（15〜20）が、それ自体はペンキではない**。この場合の判定は「halo ≥ 数千」＋「**周囲の黒への照り返し（spill）が見えるか**」を目視で確認する。std・halo とも低いときだけが真のペンキ。
- **注意**: 480pxテストと1600px heroで見え方が変わる。**測るのは hero**。ES 1.4→2.0で平均がほぼ動かない時はトーンマップのショルダーで飽和している＝ESでなく発光面積・光源W・遮蔽を疑う（#22）。
- **教訓**: 数値で当てるのは正しいが、**測る量を間違えると指標が絵を殺す**。平均・白飛び率は「光らしさ」を測れない。勾配（std）とハローを測る。

## 15. object.scale / transform_apply を最初から使わなければ #7/#7-b は起こらない（020で確立）
#7 と #7-b は「どっちに転ぶか毎回probeで確認」が必要で消耗する。020では**そもそも両方使わない**書き方にして罠を構造的に消した。
- bmesh でメッシュを**ワールド実寸**で作り、`object.location` は `(0,0,0)` のまま置く。`bmesh.ops.create_cube(size=1)` → `bmesh.ops.scale` → `bmesh.ops.translate` は**メッシュ頂点を動かすだけ**でオブジェクト変換には触れないので、transform_apply が要らない。
- アニメはその上に `o.location.z` のキーを**差分として**乗せればよい（メッシュのオフセット＋location で合成される）。
- boolean のカッターも同じ作りにして被演算子に `parent` するだけで、穴が材にぴったり固定される（親のparent_inverseは既定のidentityのまま触らない＝#9とも整合）。
- ベベル幅が非一様スケールで歪む問題も同時に消える（object.scaleが1のまま）。
- **教訓**: 新規スクリプトはこの方式を既定にする。primitive_*_add＋scale＋transform_apply は罠が多い。

## 16. 480pxでは「造形の意味的な破綻」が見えない。シルエットの"読み"はheroサイズで判定する（021で確認）

- **テーパー12%（下0.245→上0.215）** → **花瓶／ゴミ箱**に見えた（480pxでは気づかない絞り量）
- **丸い管の縦ハイライト** → **黒いプラスチックのパイプ**に見えた（後述 #17）

**教訓**: 480pxテストで判定してよいのは**構図・余白・明暗のバランス**まで。「**何に見えるか**」＝シルエットの意味は解像度が上がると変わる。
造形が固まった時点で**必ず一度 heroサイズ（1600×2000/96smp、M1で約30秒）を目視**してから本番へ。480pxを何周回しても意味的破綻は出てこない。

## 17. 丸い面は平面より環境光を拾う。ラフネス0.20+コート0.28の「黒漆」レシピは筒/球には効かない（021で確認）
010 SOROBAN で確立した黒漆レシピ（ラフ0.20 / スペキュラ0.32 / コート0.28）は**平らな枠・桁**用。
021 の丸い管に同じ値を当てたら、明るい環境光（0.92×0.55）が丸面に**縦一本の鋭い白ストリーク**として映り込み、**黒いプラスチックのパイプ**に転んだ。
平面はハイライトが面ごとに均一に乗るが、曲面は環境の映り込みが線状に集中するため、同じ値でも樹脂に見える。
- **回避（021の採用値）**: ラフネス **0.34** / スペキュラ 0.30 / コート **0.08** / コートラフ 0.20。ハイライトを縦に散らし、クリアコートの照りを落とす。
- **使い分け**: 平面主体（枠・板・角柱）→ ラフ0.20・コート0.28（010/020）。曲面主体（管・球・チューブ）→ ラフ0.34・コート0.08（021）。019 AYAの丸チューブが0.34で黒を保てていたのとも整合。

### 17-b. 第3の使い分け：紙・布は「マット」。漆レシピを紙に当てると機械部品になる（022で確認）

### 17-c. 一様に明るい env(0.92) 下では、平面の黒は「素材」でなく「鏡面反射率」で決まる（024で確認）

## 18. 4:5ポートレートの実効フレームは 縦3.52 × **横2.81**。横に広がる造形は横で死ぬ（022で確認）
シリーズ不変のカメラ（85mm・距離8.3・sensor_fit=AUTO）では、36mmのセンサーが**長辺＝縦**に載るので、
被写体が置ける実寸は **縦3.52 / 横2.81**。022（開いた本）はこれを知らずに設計して2周溶かした。
- 縦長の造形（管・柱）ばかり作ってきたので露呈しなかったが、**横に開く造形では横が先に死ぬ**。
- 開いた本の横幅 ≈ `2·(内半径+パネル幅)·sin(開き角)`。設計時にこの式を先に立て、55〜65%（≈1.6〜1.8）に収まる開き角を決める。
- **リグの回転 ROT0 は符号が効く**：カメラは x=+0.55 から x=+0.1 を見ているので**フレームは右に余裕があり左が詰まっている**。
  022では ROT0=-10° で左の表紙がカメラ側へ振り出て遠近で1.14倍に拡大し、フレーム左で切れた（右は大きく空いたまま）。
  **+10°**（＝空いている側へ振る）で解決。造形を縮める前に符号を疑う。
- **probeで数値で当てる**（レンダーで目視しない）：`bound_box` のworld x範囲を print し、近い側の y で `8.3/(8.3-|y|)` の遠近倍率を掛けて実効幅を出す。

## 19. 軸上に置いた点光源は、その軸から放射状に伸びる面を洗えない＝LEDテープになる（022で確認）
022で「綴じ目が光る」を作るのに、011 NAMI・014 NENRIN で効いた**柔らかいライム点光源**を綴じ軸に置いたら破綻した。
- **症状**: 3灯 → 見開きに光の玉が3つ＝「本の中に豆電球が3つ入っている」。9灯に増やして線光源を近似しても粒が繋がらず＝**LEDテープ**（品質チェックリストの禁則そのもの）。
- **正体**: 光源を軸に置くと、その軸から放射状に伸びる面（ページ）は**光線とほぼ平行＝グレージング入射**なので cos が効かず面が洗えない。
  光源のすぐ隣（0.07）だけが逆二乗で強く光る＝離散したホットスポットしか出ない。**灯数を増やしても解決しない**（幾何の問題なので）。
- **効いた対処（022の答え／007 KIRITORI・017 と同じ手）**: **光源を「凹面」にして、造形の奥に隠す**。
  022では後方170°を覆う背表紙シェルの**内側（凹面）だけを発光**させた。凹面はスリットの正面を向くので面で光り、
  見開きのスリット越しにだけ覗く＝「本の真ん中が光っている」に読める。**軸上に発光する棒を置く案は2回試して2回とも「裸の緑棒」**（018と同じ罠）。
- **教訓**: 011/014 で効いた手がそのまま効くとは限らない。**光源と被照射面の相対角度**（グレージングか正対か）を先に見る。
  「光に見せる」＝発光体を見せることではなく、**黒い面に減衰グラデを落とすこと**。それが出ない配置なら、配置の方を変える。

## 20. 深い容器を軸ほぼ正面から見ると「中の水位」は見えない＝上下運動はモーションに読めない（023で確認）
023（UTSUWA 器）で「器の中を光が液のように満ちて上がる」を作ろうとしたが、シリーズ固定カメラ
（z=1.95→look z=1.48＝ほぼ水平・器を約20°傾けても視線は開口面に浅く入るだけ）では、**深い器の内部の
水位は原理的に見えない**（現実の抹茶碗を上から覗いても水位が分からないのと同じ）。
- **症状**: 発光プール円盤を器ローカルzで上下させても、top-down 気味の視線ではプールが開口を常に覆い、
  「明るい抹茶／暗い抹茶」の**明滅にしか読めない**。器を筒茶碗（背高・細）に変えて内壁を縦に見せても
  同じ（開口が細くなるだけで水位は依然見えない）。低い茶碗でも同様。
- **効いた対処**: 水位＝幾何の上下運動で見せるのは諦め、**内壁の自発光を切って**（プール＋随伴点光源だけを
  内部光源にする）、`u=0.5(1-cos2πt)` でプール発光と点光源エネルギーを明滅させ、**『光で満ち→闇へ引く』の
  明暗の落差**で満ち引きを読ませた（NENRIN/NAMI と同じ静かな明滅ループ）。内壁を常時自発光させると空でも
  ライムのままで落差が出ないので、内部光源は「満ちで壁に近い／空で壁から遠い」プール随伴光にして
  幾何falloffでも差を稼ぐ。プールの上昇・拡大は glb アニメ（transform）としては残す（材質発光は glb 不可）。
- **形は美しさで選ぶ**: モーションが同じなら、**抹茶茶碗（低く広い chawan）**は筒茶碗より一目で茶道の器＝
  ブランド（黒×抹茶）に読める。読み（1秒で何か）が強い方を採る。
- **教訓**: 「容器の中で何かが上下する」演出は、開口を横〜斜めから見下ろせる浅い器か、断面を見せる構造
  （007/017 の窓・鍵穴）でないと読めない。深い器は**中身の明暗**でしか語れない。

## 21. 掃引した螺旋/線状リッジは両端を必ず細らせる（ブラント端キャップは機械部品に見える）（025で確認）
025（UZU 渦）で、対数螺旋の腕を一定断面の掃引チューブで作ったら、**端の平らなキャップが切った角材のように見え**、
渦の「流れ」でなく機械部品の羅列に転んだ（heroで発覚・#16）。
- **回避**: 掃引断面のスケールを両端で細らせる taper を掛ける。u=m/M（0..1）に対し
  `taper = 0.06 + 0.94·max(0, min(u/0.20, (1-u)/0.20, 1))`＝端20%でランプ・中央は満幅・端は6%まで痩せる。
  これで端キャップが極小になり、腕が水流の筋のように**フェザー**して一気に「渦」に読めた。
- **応用**: 線・筋・光跡・流れを掃引で表す時は端を必ず細らせる（006 ITOの bevel_factor 端消しと同じ発想を断面スケールで）。

### 20-c. 「容器の中の上下」は深い器では読めないが、回転モチーフは浅くすると中心が視覚中央に来る（025で確認）
#20 は「深い器を軸正面から見ると水位が見えない」だった。025 では逆向きの同じ幾何が効いた：
**深いボウル（BOWL_D=0.40）の最深部＝発光の眼が、傾け（TILT）で視覚的に下へ沈み**、「光が中心でなく下部に溜まる」絵になった＝
シリーズの核「真ん中に光」が壊れる。
- **回避**: 上下運動（満ち引き）で語らない**回転/放射モチーフ**なら、器を**浅い螺旋ディスク（BOWL_D=0.22）＋やや強めの TILT(36°)** にして、
  中心の光を視覚中央に保つ。浅くしても腕/リッジの陰影で立体感は出る（深さより回転の流れが主役）。
- **判定**: 「中心の光は画面の中央に来ているか」を hero で必ず見る。深さは"満ち引きを見せたい時だけ"必要で、回転モチーフでは中心が沈むデメリットの方が大きい。

## 22. 発光は「面積」で稼ぎ「強度」で稼がない。細い発光体はコアが白緑に飛ぶ（026で確認）
026（SUNADOKEI 砂時計）で、くびれを落ちる細い発光体（半径0.030）の色が hero で **#ADD859／コア #DBF996**（青が浮いた白緑）に転んだ。
#14 に従い ES を 2.6→1.6→0.95→0.65 と振ったが、**平均は動くのにコアは #DBF996 のまま動かなかった**＝犯人は emission ではない。
- **正体は2つ**: ①至近（数cm）に置いた**点光源が強すぎる**（42W）と、その周りの面が近距離で焼けて白緑のコアを作る。②**発光体が細い**と、少ない画素に輝度が集中してトーンマップのショルダーに乗り、色が必ず白へ寄る。
- **効いた対処**: 点光源 42W→**14W**、発光体を細く強く→**太く弱く**（半径 0.030→0.050／ES 1.6→1.25）。この2手で avg **#A7DF34**（目標 #A5E02E 一致・白飛び0%）に一発で入った。lime画素数も 2.4k→4.6k に増え、絵の中の「光の存在感」も同時に上がる。
- **教訓**: 色が合わない時は ES を振る前に「**発光している面積**」と「至近の点光源のW数」を疑う。#14 の測定は**平均だけでなくコア（上位20%）も併記**すると、emission が犯人か光源が犯人かを1回の測定で切り分けられる。

## 23. 「容器＋中身」は容器のプロファイルを**直線**にすると、中身の増減が scale だけで書ける（026で確認）
026 で最初、砂時計のバルブを正弦で膨らませた（球状バルブ）ら hero で **黒いスツール／鳥籠**に転んだ（#16）。**直線の双円錐（▽△）＝砂時計のアイコン**に変えたら一発で読めた。
さらに直線にした副産物が大きい：**直円錐の「水位から下の残り」は元の円錐と相似形**なので、シェイプキーもメッシュ再構築もなしに **object.scale のアニメだけ**で中身の減り／積もりが書ける（glb にも transform として乗る）。
- **一様スケールだけでは足りない**：減った砂が器の壁から離れて「宙に浮いた黒い塊」に見える。上面を壁に接地させる径倍率は幾何から出せる＝`fx(s) = (R_NECK + s·(R_MAX−R_NECK)) / R_MAX`（目分量で決めない）。積もる側は安息角一定＝**一様スケールが物理的に正しい**ので、上下で法則を変えてよい（満杯 s=1 で両者一致するのでループの端は閉じる）。
- **応用**: 器・容器ものは「曲線で美しく」より先に「**中身の運動が数式で書ける形か**」でプロファイルを選ぶと、実装量と破綻の両方が減る。

### 23-b. 反転（180°フリップ）ループの成立条件（026で確立）
013/025 の「N-fold 対称を整数回転で自己一致させる」の**反転版**。カメラ光軸＝Y軸まわりに 180° 回すと、画面内で「ひっくり返す」動作に読める（縦長 4:5 でも横幅は対角 √(H²+W²) だけ要るので #18 の 2.81 に収まるか先に計算する）。
- 閉じる条件は「**反転後の状態＝初期状態**」。①造形が上下対称であること、②周方向の要素は **φ→π−φ に閉じる角度集合**であること（`2πi/N` は **N が偶数**なら閉じる。奇数本を使うなら 90°/210°/330° のように y 軸対称に並べる）、③中身の両極端（満杯 / 空）が互いの鏡像であること。
- 残差（空の側に残す最小量 S_MIN）は上下で法則が違うと厳密な鏡像にならないので、**目に見えない大きさ**にしておく（026 は 0.06）。

## 24. 【最重要・退行事例】数値基準の最適化がライムを「光」から「ペンキ」に退化させた（016/020〜026・2026-07-21に発覚）
シリーズ後半（016 と 020〜026 の8作）で、ライムが**縁のくっきりした均一なベタ塗り**＝緑のシール／ネオンプラスチックに退化していた。各作は当時の基準（#11「輝点を作らない」・#13「純発光体」・#14旧版「平均一致・白飛び0%」）を**全て満たして「合格」しながら**、絵として最重要の「光っている」が死んでいた。ユーザー（Ryota）の指摘で発覚。
- **メカニズム**: 3つのルールの複合。#11がブルームの種となる輝点を締め出し → #13が反射のきらめきを消し → #14旧版が「白飛び0%」を強制して Bloom 閾値(1.2)を超える画素を根絶 → **ハローが物理的に出ない**。指標が目的化する典型（Goodhartの法則）。
- **実測**: 健全な旧作は lime 輝度std 40〜66・ハロー画素 数千〜6万。退化した8作は std 11〜16（024/026でも32〜38）・ハローほぼ0。
- **回避**: #14改訂版の3条件（**中間調=#A5E02E域・std≥35・halo≥1000**）を hero で測る。#11 は「輝点を作らない」ではなく「**発光面積を広く・減衰を柔らかくして箱アーティファクトを回避する**」と読み替える（011自身が成功した手。輝点の全面禁止はブルームの禁止と同義になってしまう）。
- **教訓**: **数値チェックに通っても、最後に必ず 002 OBI と並べて「これは光か、塗装か」を目視で問う。** 基準を新設・改訂した直後の数作は、その基準が絵を殺していないか疑うこと。
